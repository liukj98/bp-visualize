## Context
这是一个全新的前端项目，旨在通过交互式可视化帮助用户理解BP反向传播算法。项目参考知乎文章（zhihu p/40378224，Matt Mazur 经典BP教程），使用其中的具体数值案例作为默认演示场景。项目需要同时处理数学计算和图形渲染，对性能和可读性有较高要求。

## Goals / Non-Goals
- Goals:
  - 提供流畅的神经网络可视化动画（≥30fps）
  - 支持用户交互控制训练过程（单步/自动、参数调节）
  - 清晰展示BP算法的四个核心阶段
  - 在教育性和交互性之间取得平衡
- Non-Goals:
  - 不支持大规模网络（>100节点）的训练
  - 不实现完整的深度学习框架功能
  - 不支持 CNN/RNN 等复杂网络结构
  - 不需要后端服务或数据持久化

## Decisions

### 1. 渲染技术选择：Canvas
- **决定**: 使用 HTML5 Canvas 作为主要渲染技术
- **原因**: Canvas 在处理大量节点和连线的动画时性能优于 SVG，且更适合自定义绘制流动效果
- **替代方案**: SVG（更易做DOM交互但大量元素性能差）、WebGL（过度复杂）

### 2. 架构模式：MVC + 事件驱动
- **决定**: 采用 MVC 模式，Model 处理纯数学计算，View 负责 Canvas 渲染，Controller 管理用户交互
- **原因**: 关注点分离，便于独立测试算法逻辑和调整视觉表现
- **替代方案**: 单体架构（简单但难维护）、框架驱动（React/Vue 对 Canvas 密集操作增加复杂度）

### 3. 动画系统：requestAnimationFrame + 状态机
- **决定**: 使用 requestAnimationFrame 驱动渲染循环，用状态机管理动画阶段（前向传播→损失计算→反向传播→权重更新）
- **原因**: 原生API性能最优，状态机确保动画阶段有序执行
- **替代方案**: CSS 动画（不适用于 Canvas）、定时器（帧率不稳定）

### 4. 构建工具：Vite
- **决定**: 使用 Vite 作为开发/构建工具
- **原因**: 开发体验优秀（HMR快速）、配置简洁、原生支持 ES 模块
- **替代方案**: Webpack（配置复杂）、无构建工具（不支持模块化开发）

### 5. 数学公式渲染：KaTeX
- **决定**: 使用 KaTeX 渲染数学公式
- **原因**: 渲染速度快（比 MathJax 快很多）、体积小、无外部依赖
- **替代方案**: MathJax（功能更全但体积大且慢）、图片（不灵活）

## Risks / Trade-offs
- **Canvas 交互限制** → 需要手动实现点击检测（如节点hover/click），通过坐标计算解决
- **大网络性能** → 限制最大网络规模（4层×8节点），超出时给出提示
- **数值精度** → JavaScript 浮点数精度有限，对教学演示影响不大
- **浏览器兼容** → Canvas API 在现代浏览器中支持良好，风险低
